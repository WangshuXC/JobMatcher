## 探测字节跳动招聘 API

### 关键发现
字节跳动招聘也有 JSON API，且**不需要签名**，可以直接从服务端调用！

### 列表 API（POST）
```
POST https://jobs.bytedance.com/api/v1/search/job/posts
Content-Type: application/json

{
  "keyword": "前端",
  "limit": 10,       // 最大支持 50
  "offset": 0,       // 分页偏移
  "job_category_id_list": ["6704215862603155720"],  // 研发大类
  "tag_id_list": [],
  "location_code_list": [],
  "subject_id_list": [],
  "recruitment_id_list": [],
  "portal_type": 2,
  "job_function_id_list": [],
  "storefront_id_list": [],
  "portal_entrance": 1
}
```

返回字段：
- `data.count` - 总数（如 653）
- `data.job_post_list[].id` - 职位 ID
- `data.job_post_list[].title` - 标题
- `data.job_post_list[].description` - **完整职责描述**
- `data.job_post_list[].requirement` - **完整职位要求**
- `data.job_post_list[].city_info.name` - 城市
- `data.job_post_list[].city_list[]` - 多城市列表
- `data.job_post_list[].job_category.name` - 类别（如"前端"）
- `data.job_post_list[].job_category.parent.name` - 父类（如"研发"）
- `data.job_post_list[].recruit_type.name` - 类型（如"正式"）
- `data.job_post_list[].publish_time` - 发布时间戳
- `data.job_post_list[].code` - 职位编号

### 详情 API（GET）
```
GET https://jobs.bytedance.com/api/v1/job/posts/{postId}?portal_type=2&with_recommend=true
```
但因为列表 API 已包含完整 description 和 requirement，**实际不需要调详情 API**。

### 关键优势
- ✅ 不需要 `_signature`，直接调用
- ✅ 列表 API 返回完整数据（description + requirement），无需抓详情页
- ✅ `limit` 最大支持 50
- ✅ 比 Playwright 解析 DOM 快得多且更稳定
- category ID `6704215862603155720` = 研发大类

## 修复阿里巴巴爬虫 + 增量实时推送

### 问题1：阿里 API 被 baxia 反爬拦截
在 `page.evaluate` 中手动调用 `POST /position/search` 始终返回 `totalCount=0`，虽然页面上能正常显示 1134 个职位。

**根因**: baxia 安全组件会检测请求的安全上下文，`page.evaluate` 中的 `fetch` 缺少必要的安全标记。

**解决方案**: 改用 **Playwright `page.route()` 拦截页面自身的 API 请求和响应**。
- `setupRouteInterceptor()`: 拦截 `**/position/search**` URL，使用 `route.fetch()` 获取响应后解析并传递回页面
- `initSubsite()`: 设置拦截器后导航到子站列表页，等待初始 API 响应
- `clickNextPage()`: 通过 DOM 点击分页按钮触发翻页
- 每次翻页自动拦截新的 API 响应数据
- 删除了旧的 `fetchSearchApi()` 和 `window.__sysconfig` 依赖

### 问题2：抓取完成后切换到职位列表看不到新数据
原 `/api/crawl` 是同步 POST，等全部抓完才一次性返回和存储。

**解决方案**: 改为 **SSE (Server-Sent Events) 流式返回 + 增量存储**
1. `src/lib/crawler/base.ts`:
   - 添加 `onJobsBatch?: (jobs: JobPosting[]) => void` 回调
   - `crawlDetailsInParallel` 每批完成后调用回调
2. `src/app/api/crawl/route.ts`:
   - 改为 SSE 流式响应 (`text/event-stream`)
   - 事件类型: `jobs`(新数据) / `progress`(进度) / `result`(单源完成) / `done`(全部完成) / `error`
   - 设置 `crawler.onJobsBatch` 回调：每批完成立即 `jobStore.upsertMany()` 存储 + SSE 推送
3. `src/components/CrawlerPanel.tsx`:
   - 使用 `ReadableStream` 消费 SSE 流
   - 每收到 `jobs` 事件就调用 `onJobsUpdate?.()` 通知父组件
   - 实时显示进度信息和已入库职位数
4. `src/app/page.tsx`:
   - 添加 `handleJobsUpdate` 回调传给 CrawlerPanel
   - 每批新数据到达时触发 `refreshTrigger++`，刷新 JobList
5. `src/lib/store.ts` (无需改动):
   - `upsertMany()` 已按 `source_sourceId` 作为 Map key 实现增量去重覆盖

## 修复：抓取数据时 location 筛选失效

### 问题
在爬虫抓取数据期间，如果用户已选择了 location 筛选，新数据到达时筛选会被重置为"全部"。

### 根因
`JobList.tsx` 中 `useEffect` 监听 `refreshTrigger` 变化时，调用 `fetch("/api/jobs")` 不带任何筛选参数，导致每次新数据到达（`onJobsUpdate` → `refreshTrigger++`）时，返回的是全量数据，覆盖了用户当前的筛选结果。

### 修复
在 `JobList.tsx` 中新增 `filtersRef`（useRef）来保存当前筛选条件（selectedSource、keyword、selectedLocation）。`useEffect` 中通过 `filtersRef.current` 读取最新筛选条件构建请求参数，而不是不带参数请求全量数据。`useEffect` 依赖列表仅保留 `refreshTrigger`，避免与用户手动筛选操作（`fetchJobs`）产生双重请求。

## 修复：阿里巴巴爬虫子站初始化全部失败

### 问题
阿里巴巴爬虫在初始化各子站时，所有子站均报"初始数据获取失败"。

### 根因（两层 bug）

**Bug 1: `route.fetch()` SSL 错误**
原实现使用 `page.route()` + `route.fetch()` + `route.fulfill()` 模式拦截 `/position/search` API 响应。`route.fetch()` 使用 **Node.js 网络栈**重新发起请求（而非浏览器内部网络栈），部分子站（尤其是淘天集团 talent.taotian.com）的服务器存在 TLS 版本兼容问题，导致 SSL 握手失败：
```
EPROTO ssl3_get_record: wrong version number
```

**Bug 2: 响应监听器的竞态条件**
将 `page.route()` 改为 `page.on('response')` 后又遇到竞态：原 `waitForResponse()` 方法在被调用时立即将 `resolveResponse` 重新指向新 promise 的 resolve 函数。但因为 handler 是异步的（`await response.text()`），handler 中的 `resolveResponse(data)` 执行时 `resolveResponse` 已指向新的 resolve，数据被 resolve 到了没人等待的 promise 上。

### 修复

**`src/lib/crawler/sources/alibaba.ts`**:
1. 将 `page.route()` + `route.fetch()` + `route.fulfill()` 拦截模式改为 `page.on('response')` **被动监听模式**，不干预原始请求/响应流，彻底规避 SSL 兼容性问题
2. 监听器改为**队列模式**（生产者-消费者）：
   - handler 作为生产者：解析响应后，若有等待中的 `pendingResolve` 则直接 resolve；否则入队 `dataQueue`
   - `waitForResponse()` 作为消费者：若队列有数据则立即返回；否则注册 `pendingResolve` 等待
   - 彻底消除竞态条件，无论 handler 先于还是后于 `waitForResponse` 调用都能正确工作
3. `cleanup()` 从 `async` 改为同步（`page.off` 是同步的）
4. 接口从 `RouteInterceptor` 重命名为 `ResponseListener`，变量从 `interceptor` 改为 `listener`

### 验证结果
所有 6 个子站（+高德、千问）均可成功初始化：
- 阿里云: 1134 jobs ✅
- 淘天集团: 859 jobs ✅（之前因 SSL 错误必定失败）
- 阿里国际数字商业: 511 jobs ✅
- 菜鸟集团: 132 jobs ✅
- 钉钉: 147 jobs ✅
- 通义实验室: 42 jobs ✅

## 修复：阿里巴巴爬虫 `positions is not iterable` 崩溃

### 问题
`crawlJobList` 第 336 行直接将 `initialData!.content!.datas` 传入 `processPositions`，但某些子站的 API 返回中 `datas` 可能为 `undefined`/`null`（非数组），导致 `for...of` 抛出 `TypeError: positions is not iterable`，整个爬虫崩溃。

## 修复：爬虫中断时已爬取的数据不会被保存

### 问题
当爬虫中途报错（如 `positions is not iterable`），所有已爬取的数据全部丢失。前端显示"爬取失败，0 个职位"。

### 根因（三层问题）

**1. `base.ts` 的 `crawl()` 方法：一个异常全盘崩溃**
`crawlJobList()` 和 `crawlDetailsInParallel()` 共享同一个 try-catch，任一抛异常都会跳到 catch，返回 `jobs=[]`（空数组）+ `status="error"` + `jobCount=0`。已通过 `onJobsBatch` 增量保存的数据虽然在 jobStore 中，但 result 事件上报 0，前端显示失败。

**2. 字节跳动/腾讯爬虫：crawlJobList 不调用 onJobsBatch**
这两个爬虫在 `crawlJobList` 内部**不调用 `onJobsBatch`**，完全依赖后续的 `crawlDetailsInParallel`。如果 `crawlJobList` 中途失败（如网络中断），`crawlDetailsInParallel` 根本不会被执行，已爬取的列表数据全部丢失。

**3. 阿里巴巴爬虫：一个子站异常炸掉全部**
`crawlJobList` 的子站循环体没有 try-catch，一个子站的异常会直接冒泡到 `base.ts` 的 catch，导致后续所有子站都不会被执行。

### 修复

**`src/lib/crawler/base.ts`**：
1. `crawlJobList()` 和 `crawlDetailsInParallel()` 分别独立 try-catch
2. `crawlJobList` 中途失败时，已获取的 partialJobs 仍然传给 `crawlDetailsInParallel` 继续处理
3. 通过包装 `onJobsBatch` 回调，跟踪已通过增量推送保存的 `pushedJobCount`
4. `jobCount` 使用 `Math.max(jobs.length, pushedJobCount)`，确保已推送的数据被正确统计
5. 有错误但有数据时 status 报 `"completed"`（而非 `"error"`），前端不会误显示全部失败

**`src/lib/crawler/sources/alibaba.ts`**：
1. 在子站 for 循环体内包裹 try-catch，catch 中打印错误并 continue 到下一个子站
2. 确保异常时 listener 被正确清理（避免内存泄漏）
